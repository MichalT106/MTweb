Detection of Formal Errors in Final Theses using XML Technologies

🎓

Diploma thesis topic: Detection of Formal Errors in Final Theses using XML Technologies .

The goal was to bring repeatable, automation-friendly validation to LaTeX theses by converting sources into structured XML and evaluating an extensible set of rules in CI/CD and locally.

Project overview

This project implements a toolchain for detecting formal and stylistic issues in LaTeX academic theses. Instead of relying on manual PDF review or generic text linters, it converts the thesis into a structured XML representation and runs rule-based checks over the document structure.

It is designed to run automatically in GitLab CI pipelines (with artifacts and optional GitLab Pages publishing) and locally via Docker using the same conversion and detection stack.

Architecture (converter → detector → outputs)

  • Converter (`converter/`): Perl-based pipeline around the LaTeXML API that converts LaTeX sources into LaTeXML XML and injects source-location metadata for traceability back to the original `.tex`/`.bib` files.
  • Detector (`detector/`): Python rule engine built on SaxonC (XPath 3.1). It loads versioned XML rules, evaluates them over the document structure, and renders TXT/console/HTML reports.
  • CI integration (`ci/`): GitLab CI templates + helper scripts that run the same converter → detector flow inside a prebuilt Docker image and publish artifacts and optional GitLab Pages.
  • Local runner (`local_run/`): Docker Compose + wrapper scripts for reproducible local runs (single thesis or batch mode) that match CI behavior.

Workflow (from LaTeX sources to actionable findings)

  • Preprocess sources: inject `srcroot` and `srcloc` markers into `.tex`/`.bib` inputs so every finding can be traced back to the original file and line.
  • Convert with LaTeXML (Perl API): produce `thesis.xml` and, when a bibliography is declared, `thesis.bib.xml`.
  • Evaluate XPath rules (SaxonC): apply a versioned rule set to the XML structure; support both node-based checks (precise locations) and threshold-style rules.
  • Generate outputs: write TXT reports, console-friendly summaries, and optional HTML dashboards with deep links to source code.

Key features

  • Structural validation based on LaTeXML XML (document-aware checks, not just text linting).
  • Rules defined as versioned XML files (thesis + bibliography scopes) with XPath selectors and messages.
  • Central configuration to enable/disable rules and override severities (`detector/rules_config.xml`).
  • Human-friendly outputs: TXT for logs and HTML reports with an index for browsing.

Rule engine & extensibility

Checks are authored as standalone XML rule files and evaluated as XPath 3.1 expressions over the LaTeXML document model. This keeps most new checks “data-driven”: adding rules typically does not require Python changes.

  • Two scopes: thesis rules (`detector/rules/thesis_rules/`) and bibliography rules (`detector/rules/bibliography_rules/`).
  • Central policy in `detector/rules_config.xml`: disable specific rule IDs and override severities per scope (INFO/WARNING/ERROR).
  • Rules can emit findings with precise source mapping by leveraging converter-injected `srcloc` markers embedded in the produced XML.

Outputs & results

Runs generate converter outputs (e.g., `thesis.xml`, optional `thesis.bib.xml`) and detector reports under `output/` as TXT and optionally HTML. Reports contain rule identifiers, severity, an explanation, and source pointers that can become clickable links in GitLab CI.

These artifacts can be used as a review aid, a CI quality gate (e.g., fail builds on ERROR-level findings), and an auditable record of automated thesis validation.

Reporting & navigation (findings → source code)

  • HTML reports include per-finding deep links: in GitLab CI they point to `/-/blob//#L` so reviewers can jump straight to the affected line.
  • Local runs generate editor-friendly links (`vscode://file/...`) and also provide plain `file://` links plus a short source snippet for quick context.
  • Path mapping is handled automatically for local Docker runs so reports show host paths even when the detector runs inside a container.

CI/CD integration (GitLab)

The checker is reused across multiple LaTeX repositories via templates in `ci/`. The analyzed repository includes a `.gitlab-ci.yml` based on these templates and runs the checker inside a Docker image that already contains the full toolchain under `/formal_checker`.

The pipeline bundles outputs as artifacts and can publish HTML reports through GitLab Pages for convenient browsing and review.

Packaging & distribution (reusable Docker image)

The full toolchain is packaged into a Docker image containing LaTeXML + TeX Live and the Python detector runtime. In GitLab, the checker repository builds and publishes commit-addressable image tags, so analyzed projects can pin an exact version for reproducible results.

Local usage (Docker)

A Docker runner mirrors the CI environment to avoid differences between local runs and pipeline runs. Wrapper scripts support both single-thesis and batch processing modes.

Rule extensibility

Most checks can be added without changing Python code: create a new XML rule file in the appropriate rules folder (thesis or bibliography), define an XPath selector and message, and optionally tune behavior in `rules_config.xml`.
